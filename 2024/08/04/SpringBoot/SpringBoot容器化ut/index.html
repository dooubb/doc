发表于 2024-08-04 分类于 SpringBoot 阅读次数： 本文字数： 7.7k 阅读时长 ≈ 7 分钟

单元测试UT

本地

开发在自己电脑上进行开发和测试的时候，maven几乎都是安装包或者idea自身提供的，然后其他开发和测试依赖的组件，例如mysql数据库，redis缓存等，都是使用容器化方式部署到本地，这样在开发和测试可以快速debug。

线上

线上都是流水线一体化作业，可以使用git action，git lab cicd，azure devops，jenkins等。

这些的编译，单元测试，构建服务镜像都是使用容器化来编排操作的，因为容器化是无状态的，可以快速扩展和升降级相关组件版本。

UT环境搭建

本地

在本地启动所需组件可以是容器启动，也可以是直接安装对应组件的客户端；

由于拥抱未来，所以只讨论容器启动所需组件。在容器的前提下，容器的启动方式可以是动态启动的，也可以是提前准备好的容器。

提前准备

使用docker来启动相关的mysql和redis的，如果需要保留测试数据，需要再启动的时候挂载目录来持久化；可以使用docker run也可以使用docker compose来拉起容器。

动态启动

Java有相关类库来支持UT启动所需容器，testcontainers。在UT阶段，testcontainers可以动态启动所需容器，他会自动hook本机的docker sock，这样就可以直接在本机上动态启动容器了，如果本机没有docker，那UT将无法执行。

使用方法，添加maven 中的pom依赖和对应的测试数据库驱动，我这里使用的mysql9分库分表的。

在UT的中添加启动所需组件的代码和相关初始化配置

启动UT然后打个断点，然后查看本机上启动的容器

其中有两个容器是testcontainers的，剩下的redis和mysql就是UT测试所需的组件。mysql对外暴露的端口也是33064，这样就可以使用数据库客户端进行链接了。但是线上无法固定端口，因为线上可能有多个服务同时启动UT。

线上

线上都是流水线可能使用到git action，git lab cicd，azure devops，jenkins等。所以这里直接跳过流水线执行部分，直接讨论UT启动方式。

不依赖相关组件

可以直接使用maven容器来直接进行UT.

依赖相关组件

依赖相关组件的话，启动相关组件就可以有很多方式。

docker

可以再流水线中直接docker run，比较蠢，并且多服务一起UT时会相互影响。不推荐。

docker compose

docker compose来启动，默认会创建当前docker-compose.yml所在的目录名字的网络，这样对于多个服务一起UT不会有影响，但是会有性能开销，空间换时间。

docker compose可以直接启动maven，redis，mysql等，然后将项目内容挂载到maven容器中，容器中的初始化化数据库可以使用dockerfile来自定义或者直接docker commit来完成

Java类库

对于不同的流水线触发maven指定ut的触发方式可能不同。所以测试只考虑使用docker compose来启动，并将带有testcontainers的代码挂载到maven中，执行UT

docker compose启动maven编译后，可以看到本机上的容器，testcontainers会通过挂载给maven容器的本机docker的sock来启动代码中引用的mysql和redis容器
